# Popover migration guide

Replace the `<sp-overlay>` + `<sp-popover>` composition with a single `<swc-popover>` wired to its trigger through `for` (or `triggerElement`) and `open`; the separate overlay element and trigger plumbing are gone.

## Installation

```bash
# Remove
yarn remove @spectrum-web-components/popover @spectrum-web-components/overlay

# Add
yarn add @adobe/spectrum-wc
```

Update your import:

```js
// Before
import '@spectrum-web-components/overlay/sp-overlay.js';
import '@spectrum-web-components/popover/sp-popover.js';

// After
import '@adobe/spectrum-wc/components/popover/swc-popover.js';
```

> `@adobe/spectrum-wc` is a monolithic package. Importing via subpath (e.g. `@adobe/spectrum-wc/components/popover/swc-popover.js`) registers and loads only that component's bundle.

## What changed

### Renamed

| Area            | Spectrum 1                                             | Spectrum 2                                                   |
| --------------- | ------------------------------------------------------ | ------------------------------------------------------------ |
| Tag             | `sp-popover`                                           | `swc-popover`                                                |
| Import path     | `@spectrum-web-components/popover/sp-popover.js`       | `@adobe/spectrum-wc/components/popover/swc-popover.js`       |
| Trigger wiring  | `<sp-overlay trigger="id@click">` wrapping the popover | `for="id"` (or `triggerElement`) + `open` on `<swc-popover>` |
| Open transition | `sp-opened` / `sp-closed` events (from `<sp-overlay>`) | `swc-after-open` / `swc-after-close`                         |

### Added in Spectrum 2

| Addition                                                        | Notes                                                                                                          |
| --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `for` / `triggerElement` / `manual`                             | Associate a trigger by id, by reference, or opt out of the built-in click-to-toggle                            |
| `open` (drives visibility directly)                             | Toggle the popover with one property; no overlay element required                                              |
| `modal` boolean attribute                                       | Opt in to blocking `<dialog>.showModal()`: focus trap, page inert, scroll lock                                 |
| `accessible-label`                                              | Forwarded as `aria-label` to the internal dialog surface; required in both modal and default mode              |
| `size` (`s` / `m` / `l`)                                        | Fixed inline size (336px / 416px / 576px); unset fits content                                                  |
| `offset` / `cross-offset` / `should-flip`                       | Main-axis and cross-axis nudges; opt out of flip-on-collision                                                  |
| `swc-open` / `swc-after-open` / `swc-close` / `swc-after-close` | `swc-close.detail.source` reports `'escape'`, `'outside'`, or `'programmatic'`                                 |
| Automatic trigger ARIA                                          | The component sets `aria-expanded`, `aria-controls`, and `aria-haspopup="dialog"` on the trigger in both modes |

### Removed in Spectrum 2

| Removed                                  | Replacement                                                                                                         |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `<sp-overlay>` / overlay trigger element | Wire the trigger directly with `for` (or `triggerElement`) and `open` on `<swc-popover>`                            |
| `tip` boolean attribute                  | The arrow is shown by default; set `hide-arrow` to remove it (see [step 3](#3-drop-tip-the-arrow-is-on-by-default)) |
| `<sp-dialog>` wrapper for padding        | No longer needed: `<swc-popover>` applies the correct content padding itself. Slot your content directly            |
| `sp-opened` / `sp-closed` events         | `swc-after-open` / `swc-after-close` (see [step 4](#4-update-event-listeners))                                      |
| `--mod-popover-*` custom properties      | `--swc-popover-*` custom properties; see [Styling](#styling)                                                        |

## Update your code

### 1. Replace the overlay composition

`<swc-popover>` positions itself in the top layer against the trigger, so the `<sp-overlay>` wrapper and the surrounding positioned container are no longer needed. Point the popover at its trigger with `for` and control visibility with `open`.

```html
<!-- Before -->
<sp-button id="trigger">Account</sp-button>
<sp-overlay trigger="trigger@click" type="auto" placement="bottom">
  <sp-popover>Anchored content.</sp-popover>
</sp-overlay>

<!-- After -->
<swc-button id="trigger">Account</swc-button>
<swc-popover
  for="trigger"
  placement="bottom"
  accessible-label="Account details"
>
  Anchored content.
</swc-popover>
```

When the trigger lives in a different shadow root, set the `triggerElement` property to the element reference instead of `for`.

If your content was wrapped in `<sp-dialog>` only to get padding, drop the wrapper: `<swc-popover>` applies the correct content padding itself.

### 2. Replace `type="modal"` with the `modal` attribute

A 1st-gen `type="modal"` overlay maps to the `modal` attribute. Modal mode opens a `<dialog>` with `showModal()`: focus is trapped, the page behind is inert and scroll-locked. Name it with `accessible-label`.

```html
<!-- Before -->
<sp-overlay trigger="trigger@click" type="modal">
  <sp-popover>Settings</sp-popover>
</sp-overlay>

<!-- After -->
<swc-popover for="trigger" modal accessible-label="Account settings">
  Settings
</swc-popover>
```

### 3. Drop `tip`; the arrow is on by default

The arrow (tip) now renders by default. Remove the `tip` attribute. If you intentionally rendered a popover with no arrow, add `hide-arrow`.

```html
<!-- Before (arrow shown only with tip) -->
<sp-popover tip>Anchored content.</sp-popover>
<!-- After (arrow is the default) -->
<swc-popover for="trigger" accessible-label="Anchored content">
  Anchored content.
</swc-popover>

<!-- Before (no arrow) -->
<sp-popover>Anchored content.</sp-popover>
<!-- After (opt out explicitly) -->
<swc-popover for="trigger" hide-arrow accessible-label="Anchored content">
  Anchored content.
</swc-popover>
```

### 4. Update event listeners

Listen on the popover itself. `sp-opened` / `sp-closed` become `swc-after-open` / `swc-after-close` (fired after the transition). To know why a popover closed, read `swc-close.detail.source`.

```js
// Before
overlay.addEventListener('sp-opened', onOpened);
overlay.addEventListener('sp-closed', onClosed);

// After
popover.addEventListener('swc-after-open', onOpened);
popover.addEventListener('swc-close', (event) => {
  // event.detail.source is 'escape' | 'outside' | 'programmatic'
});
popover.addEventListener('swc-after-close', onClosed);
```

## Accessibility

- The component sets `aria-expanded`, `aria-controls`, and `aria-haspopup="dialog"` on the resolved trigger automatically, in both modes. Remove any of this wiring you previously did by hand through the overlay.
- The popover surface is a dialog in both modes, so it must have an accessible name: set `accessible-label` on every `<swc-popover>`, modal or not. The component dev-warns when opened without one. See [step 2](#2-replace-typemodal-with-the-modal-attribute).
- Give the trigger a clear, visible label; the popover does not name its own trigger.
