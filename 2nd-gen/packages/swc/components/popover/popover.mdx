## Anatomy

A `<swc-popover>` renders an anchored top-layer surface that wraps slotted
content and, by default, an arrow (tip) pointing at its trigger.

_Anatomy story and full part breakdown to come._

## Options

`placement`, `size`, `hide-arrow`, `offset`, `cross-offset`, and `should-flip`
tune positioning and appearance. `modal` opts into blocking `<dialog>` behavior
(focus trap, background inert, `role="dialog"`); pair it with `accessible-label`
to name the dialog (see [Accessibility](#accessibility)).

_Option stories to come._

## Behaviors

### Custom anchor

The toggling control and the positioning anchor can be different elements: a
button drives `open` while the popover anchors to another element via the
`triggerElement` property (`manual` suppresses the anchor's own click-to-toggle).

## Accessibility

The resolved trigger receives `aria-expanded` and a durable `aria-controls`
relationship wired through the element-reference IDL so it works across shadow
boundaries. `Escape` and outside-click dismiss in the default (auto) mode.

### Naming a modal popover

In modal mode the internal `<dialog>` provides `role="dialog"`, a focus trap, and
`aria-haspopup="dialog"` on the trigger. A dialog must have an accessible name, so
**set `accessible-label` whenever you set `modal`**: its value is forwarded as
`aria-label` to the internal `<dialog>`. Omitting it leaves a nameless dialog (an
authoring bug); in development the component warns when `modal` is set without an
`accessible-label`.

```html
<swc-popover modal accessible-label="Account settings" for="account-trigger">
  <!-- dialog content -->
</swc-popover>
```

`accessible-label` has **no effect in the default (auto) mode**: that surface is a
roleless container, so the slotted content and the trigger own the semantics.
