## Anatomy

The opacity checkerboard is a shared CSS utility, not a custom element. It lives at `stylesheets/_lit-styles/opacity-checkerboard.css` as an importable lit `css` fragment. A consuming component imports it directly and adds it to its `styles` array:

Apply `.swc-OpacityCheckerboard` to a decorative element rendered behind transparent or semi-transparent content. The consuming component is responsible for keeping that element out of the accessibility tree (for example `aria-hidden="true"`).

> These demos render through `demo-opacity-checkerboard-swatch`, an internal Lit host that adopts the fragment into its own shadow root exactly as a real consuming component would. The accessible name and opacity live on the host (`role="img"` + `aria-label`); the checkerboard layer is decorative and `aria-hidden`.

## Usage

1. Import the CSS file from `@spectrum-web-components//stylesheets/_lit-styles/opacity-checkerboard.css`.
2. Add the styles to your styles array.
3. Add the `swc-OpacityCheckerboard` class to the `span` or `div` element where you want the checkerboard to appear, and make sure the element is `aria-hidden="true"`.

```typescript
import { css, html, LitElement, type TemplateResult } from 'lit';
import { customElement, property } from 'lit/decorators.js';
import opacityCheckerboardStyles from '@spectrum-web-components//stylesheets/_lit-styles/opacity-checkerboard.css';
export class DemoOpacityCheckerboardSwatch extends LitElement {
  /**
   * Adopt the shared fragment into this element's shadow root, then layer the
   * component-local styles on top.
   */
  static override styles = [
    opacityCheckerboardStyles,
    css`
      :host {
        display: inline-block;
      }
      .swatch {
        position: relative;
        inline-size: 120px;
        block-size: 120px;
        border: 1px solid var(--swc-gray-300, #ccc);
        border-radius: 4px;
        overflow: hidden;
      }
      /* Decorative checkerboard and the color fill share the swatch box. */
      .swc-OpacityCheckerboard,
      .fill {
        position: absolute;
        inset: 0;
      }
      .fill {
        background: var(--demo-fill, transparent);
      }
    `,
  ];
  /** Human-readable name including opacity; lives on the control, not the pattern. */
  @property()
  public label = '';
  /** CSS color for the fill layer, e.g., `rgba(255, 0, 0, 0.4)` or `transparent`. */
  @property()
  public color = 'transparent';
  /** Checkerboard square size: medium (default, no modifier) or small. */
  @property({ reflect: true })
  public size: 'm' | 's' = 'm';
  protected override render(): TemplateResult {
    const sizeClass = this.size === 's' ? 'swc-OpacityCheckerboard--sizeS' : '';
    return html`
      <div
        class="swatch"
        role="img"
        aria-label=${this.label || 'Color preview'}
        style="--demo-fill: ${this.color}"
      >
        <!-- Decoration only: hidden from assistive technologies, never focusable. -->
        <span
          class="swc-OpacityCheckerboard ${sizeClass}"
          aria-hidden="true"
        ></span>
        <span class="fill" aria-hidden="true"></span>
      </div>
    `;
  }
}
```

## Options

### Sizes

The default square size uses the medium token. The `.swc-OpacityCheckerboard--sizeS` modifier switches to the small token, halving the square size.

## Behaviors

### Transparent content

The checkerboard is meant to sit behind transparent or semi-transparent content so the level of transparency reads against a neutral pattern. Fully opaque content covers it entirely.

## Accessibility

The checkerboard is **visual-only**. It carries no role, name, or value, is never a tab stop, and must not be the only cue for transparency. Accessibility is owned by the consuming component, not by this utility. For the full rationale and consumer patterns (swatch, color slider, color loupe, thumbnail, color handle), see the opacity checkerboard accessibility migration analysis under Contributor docs → Project planning → Components.

### Features

The shared `css` fragment sets no ARIA. It only keeps `forced-color-adjust: none` under `@media (forced-colors: active)` so the pattern stays visible in high-contrast mode. That visibility is intentional for color tools and does not make the pattern semantic for assistive technologies.

### Best practices

- **Hide the decoration.** Mark the `.swc-OpacityCheckerboard` element `aria-hidden="true"` when it is only a background ([WCAG 1.1.1](https://www.w3.org/WAI/WCAG22/Understanding/non-text-content.html)). Never put `aria-hidden` on a wrapper that also contains slotted or interactive content; it suppresses every descendant.
- **Name and value on the real control.** Expose color and opacity in human-readable text on the focusable or semantic element (`aria-label`, a visible label, or `aria-valuetext`), as the demo host does with `role="img"` and an `aria-label` such as "Selected color: red, 40% opacity" ([WCAG 4.1.2](https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html)). Avoid raw `rgba(...)` strings as the only name.
- **Not focusable.** Do not add `tabindex` to a node that only shows the pattern. Tab order and activation stay on the parent widget (swatch button, color slider, color field).
- **Convey opacity textually.** Transparency must not rely on the checkerboard alone ([WCAG 1.4.1](https://www.w3.org/WAI/WCAG22/Understanding/use-of-color.html)). Pair it with a label, value, or instruction that states the opacity.
- **Live regions are rarely needed.** When opacity changes, update the control's value or visible text first. Use `aria-live="polite"` only when nothing else announces the change; never `assertive` for routine opacity updates.
