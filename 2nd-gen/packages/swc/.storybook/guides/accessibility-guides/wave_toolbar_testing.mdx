# WAVE toolbar testing

## Overview

This guide explains how to use the [WAVE (Web Accessibility Evaluation Tool) browser extension](https://wave.webaim.org/extension/) to test accessibility in the browser, with a focus on **document heading structure** (`h1`–`h6`). It pairs practical steps for installing and using WAVE with [W3C Web Accessibility Initiative (WAI)](https://www.w3.org/WAI/) guidance so you know both **how** to test and **why** headings matter for people who use assistive technology.

WAVE is a free extension for **Chrome**, **Firefox**, and **Edge**. It evaluates the **rendered** page in your browser. No page content is sent to WAVE servers, so reporting stays private. You can use it on local files, password-protected pages, intranets, and pages with dynamic or AJAX content—anywhere the browser can load the final DOM and styles.

> **Note:** WAVE is one of several useful tools. It complements (and does not replace) [keyboard testing](./keyboard_testing.mdx), [screen reader testing](./screen_reader_testing.mdx), and manual review.

## What the WAVE extension is for

Use WAVE to get quick feedback on accessibility, including:

- **Automated and semi-automated checks** for many common issues (the **Details** tab annotates the page and summarizes errors, alerts, and features)
- **Structural review** of headings, landmarks, and how they appear in an outline (**Structure** tab)
- **Reading and tab order** for navigable elements versus visual layout (**Order** tab)
- **Color contrast** exploration and a **desaturate** view on the **Contrast** tab (see [Contrast tab and color testing](#contrast-tab-and-color-testing))

WAVE is especially helpful for **spot-checking heading levels** and **landmarks** before or alongside screen reader testing.

## Install the WAVE extension

1. Open the official WAVE extension page: [WAVE Chrome, Firefox, and Edge extensions](https://wave.webaim.org/extension/)
2. Choose the install link for your browser (Chrome Web Store, Mozilla Add-ons, or Microsoft Edge Add-ons)
3. After installation, pin the WAVE icon in the toolbar if you use it often

To allow WAVE on **local files** or in **private/incognito** windows, enable the relevant option in your browser’s extension settings for WAVE (wording varies by browser).

## Run a WAVE report

1. Navigate to the page you want to test (for example, a Storybook page or your app)
2. Click the **WAVE** icon in the browser toolbar, or use the shortcut **Control** + **Shift** + **U** (Windows/Linux) or **Command** + **Shift** + **U** (macOS). You can also use the context menu **WAVE this page** where available
3. The WAVE panel opens (often on the left). Use the tabs at the top to switch views. Click the WAVE icon again or refresh the page to close WAVE

Use the **Styles** toggle in the panel to see the page with browser styles on or off. Turning styles off can expose raw document order, which is useful with the **Order** tab.

## WAVE tabs for testing structure and more

The following sections describe the tabs that are most relevant to **heading structure**, **reading order**, and **contrast**. Screenshots show WAVE’s own extension documentation page as the example under test.

### Details tab

The **Details** tab summarizes **errors**, **alerts**, **features**, and **structural** information, and overlays icons on the live page so you can see where each item occurs.

<img
  src={waveDetailsTab}
  alt="WAVE extension Details tab: summary counts and structural elements, with h1 and h2 icons overlaid on a web page"
/>

Use it to confirm that headings are detected, that landmark regions are present, and to jump from summary rows to the exact place on the page. It is a good first stop after opening WAVE on a new screen.

### Structure tab

The **Structure** tab shows a **hierarchical outline** of headings and regions (for example, `header`, `nav`, `main`, `footer`). It helps you verify that **heading levels form a logical outline**—like a table of contents—and that they match what you intend visually.

<img
  src={waveStructureTab}
  alt="WAVE Structure tab: left sidebar lists nested h1 and h2 headings and regions; the page shows h1 and h2 badges beside matching headings"
/>

**Why this matters for headings:** Screen reader users often **navigate by heading** (see [How screen readers use headings](#how-screen-readers-use-headings)). A clear, nested outline lets users jump between sections quickly. The Structure view makes gaps, skipped levels, or missing main headings easier to see than scanning the page by eye alone.

### Order tab

The **Order** tab lists **navigable** elements in sequence with their **role** and **accessible name**, and can draw a **numbered path** on the page so you can compare **focus and reading order** to **visual placement**.

<img
  src={waveOrderTab}
  alt="WAVE Order tab: styles off, numbered arrows show tab order across links and controls on a web page"
/>

Use it to check that users who move with **Tab** and users who follow the **announced order** in a screen reader get a sequence that matches your design. It does not replace keyboard testing, but it highlights mismatches between DOM order and layout.

### Contrast tab and color testing

The **Contrast** tab lets you set **foreground** and **background** colors (including with the color picker), read the **contrast ratio**, and see **WCAG AA/AAA** pass or fail for normal or large text. WAVE also notes that **gradients**, **filters**, and **transparency** may need **manual** checks with the eyedropper or other tools.

<img
  src={waveContrastTab}
  alt="WAVE Contrast tab: foreground and background color inputs, contrast ratio 8.59 to 1, WCAG AA and AAA pass, and Desaturate page link"
/>

**Desaturate page** renders the page in **grayscale** to reduce reliance on **hue** when checking hierarchy and meaning. It can surface cases where information or structure depends too much on color. It is **not** a full simulation of all types of color vision deficiency; dedicated **color blindness** browser extensions can simulate **specific** conditions (for example, protanopia, deuteranopia) in more detail.

## Accessible heading structure (W3C WAI)

These principles align with [W3C WAI Tutorials: page structure and headings](https://www.w3.org/WAI/tutorials/page-structure/headings/) and related [WCAG 2.2](https://www.w3.org/WAI/standards-guidelines/wcag/) success criteria (see [Related WCAG success criteria](#related-wcag-success-criteria)).

### Heading ranks and nesting

- Use **native** heading elements: `h1`–`h6`
- **Rank** indicates **level** in the outline: `h1` is the top level, `h2` is a subsection of the nearest higher section, and so on
- **Organize** content so headings **reflect** the real **structure** of the page, not only font size

### Why skipping levels is usually wrong

Jumping from `h1` to `h3` without an `h2` can **break the outline** for users who navigate by heading level. They may think a subsection is missing. Fix **styling** with CSS if you need a smaller visual size while keeping the correct semantic level (for example, an `h2` that **looks** smaller).

### Exception: fixed regions (for example, sidebars)

A **sidebar** or **repeated** block that is the same on every page may use a **self-contained** mini-outline (for example, its own `h2` for “On this page”) that does not follow the main content’s next level. This is a common pattern. What matters is **consistency** and **clarity** so users are not disoriented when they move between **main** and **complementary** content. Prefer **landmarks** (`aside`, `nav`, `complementary` role) and **clear labels** so the purpose of the region is obvious.

### Sections and `aria-labelledby`

For a region with a heading, you can associate the heading with the section using `aria-labelledby` on the **sectioning** element, pointing to the heading’s `id`. That strengthens the **programmatic** relationship for assistive technology. WAVE and [semantic HTML/ARIA](./semantic_html_aria.mdx) documentation apply here; do not use heading markup **only** for visual weight.

### Main content and navigation order

The **first heading** in **main** content is often the page’s primary topic. If **site-wide navigation** and **banners** come first in the DOM, they should still be **correctly marked** (`nav`, `header`, landmarks) so users can **skip** to `main` (for example, [bypass blocks / skip links](https://www.w3.org/WAI/WCAG22/Understanding/bypass-blocks.html)). **Do not** put the main page `h1` only in a place that is not the primary content if that would misrepresent the document in the **outline**—the **Structure** tab can help you review this.

## How screen readers use headings

In **browse** (or **document**) mode, many screen readers let users move from heading to heading with a single key. For example, **NVDA** and **JAWS** on Windows use **H** for next heading; **VoiceOver** on macOS has its own heading navigation commands. That is why a **logical** `h1`–`h6` tree matters: it is a **navigation map** for the page.

For platform-specific shortcuts and testing workflow, see [Screen reader testing](./screen_reader_testing.mdx).

**Headings are for structure, not only for looks.** Use real heading elements for **section titles**; use CSS (or design system typography tokens) for **appearance**. A clear **visual** hierarchy that matches the **semantic** hierarchy helps **everyone**, including people with low vision or cognitive disabilities.

## Examples in a Spectrum Web Components context

Custom elements in this library often **compose** with content you provide in the **light DOM**. **You** are responsible for **headings and labels** around those components. Patterns that work well:

1. **One main topic per view:** A single `h1` for the page or view title, then `h2` for major sections, `h3` for subsections
2. **Component + heading:** Place a native heading **before** a block of related components so the section is named in the outline (for example, an `h2` “Actions” before a row of buttons)
3. **Avoid fake headings:** Do not use a `div` or `span` with large text when the content is a **title**; use the appropriate `h1`–`h6` and style it
4. **Dialogs and cards:** If a dialog or card has a **title**, expose it with a real heading or an appropriate **accessible name** (for example, `aria-labelledby` pointing to the title string), following the component’s documentation

Example (illustrative only; tag names may match your app’s swc elements):

```html
<header>
  <a href="#main">Skip to main content</a>
  <nav aria-label="Site">…</nav>
</header>
<main id="main">
  <h1>Project settings</h1>
  <section aria-labelledby="profile-heading">
    <h2 id="profile-heading">Profile</h2>
    <!-- swc form fields, etc. -->
  </section>
  <section>
    <h2>Notifications</h2>
    <h3>Email</h3>
    <!-- more content -->
  </section>
</main>
```

Use WAVE’s **Structure** tab on your built page to confirm that this outline matches what you ship.
