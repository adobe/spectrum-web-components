# WAVE toolbar testing

## Overview

This guide explains how to use the [WAVE (Web Accessibility Evaluation Tool) browser extension](https://wave.webaim.org/extension/) to test accessibility in the browser, with a focus on **headings** and **landmarks** in the page outline. It pairs with [Headings and landmarks](/docs/guides-accessibility-guides-headings-and-landmarks--readme) for W3C WAI principles, **landmark** and **heading** structure, how screen readers navigate by both, and Spectrum Web Components patterns—here you learn **where** in WAVE to see the outline and **how** to run a report.

WAVE is a free extension for **Chrome**, **Firefox**, and **Edge**. It evaluates the **rendered** page in your browser. No page content is sent to WAVE servers, so reporting stays private. You can use it on local files, password-protected pages, intranets, and pages with dynamic or AJAX content—anywhere the browser can load the final DOM and styles.

> **Note:** WAVE is one of several useful tools. It complements (and does not replace) [keyboard testing](/docs/guides-accessibility-guides-keyboard-testing--readme), [screen reader testing](/docs/guides-accessibility-guides-screen-reader-testing--readme), and manual review.

## What the WAVE extension is for

Use WAVE to get quick feedback on accessibility, including:

- **Automated and semi-automated checks** for many common issues (the **Details** tab annotates the page and summarizes errors, alerts, and features)
- **Structural review** of headings, landmarks, and how they appear in an outline (**Structure** tab)
- **Reading and tab order** for navigable elements versus visual layout (**Order** tab)
- **Color contrast** exploration and a **desaturate** view on the **Contrast** tab (see [Contrast tab and color testing](#contrast-tab-and-color-testing))

WAVE is especially helpful for **spot-checking heading levels** and **landmarks** before or alongside screen reader testing. Interpret what you see using [Headings and landmarks](/docs/guides-accessibility-guides-headings-and-landmarks--readme) so you know which levels, regions, and patterns to aim for in your app.

## Install the WAVE extension

1. Open the official WAVE extension page: [WAVE Chrome, Firefox, and Edge extensions](https://wave.webaim.org/extension/)
2. Choose the install link for your browser (Chrome Web Store, Mozilla Add-ons, or Microsoft Edge Add-ons)
3. After installation, pin the WAVE icon in the toolbar if you use it often

To allow WAVE on **local files** or in **private/incognito** windows, enable the relevant option in your browser’s extension settings for WAVE (wording varies by browser).

## Run a WAVE report

1. Navigate to the page you want to test (for example, a Storybook page or your app)
2. Click the **WAVE** icon in the browser toolbar, or use the shortcut **Control** + **Shift** + **U** (Windows/Linux) or **Command** + **Shift** + **U** (macOS). You can also use the context menu **WAVE this page** where available
3. The WAVE panel opens (often on the left). Use the tabs at the top to switch views. Click the WAVE icon again or refresh the page to close WAVE

Use the **Styles** toggle in the panel to see the page with browser styles on or off. Turning styles off can expose raw document order, which is useful with the **Order** tab.

## WAVE tabs for testing structure and more

The following sections describe the tabs that are most relevant to **heading structure**, **reading order**, and **contrast**. Screenshots show WAVE’s own extension documentation page as the example under test.

### Details tab

The **Details** tab summarizes **errors**, **alerts**, **features**, and **structural** information, and overlays icons on the live page so you can see where each item occurs.

  <img

  <figcaption>

    and h2 icons overlaid on a web page
  </figcaption>
</figure>

Use it to confirm that headings are detected, that landmark regions are present, and to jump from summary rows to the exact place on the page. It is a good first stop after opening WAVE on a new screen.

### Structure tab

The **Structure** tab shows a **hierarchical outline** of headings and regions (for example, `header`, `nav`, `main`, `footer`). It helps you verify that **heading levels form a logical outline**—like a table of contents—and that they match what you intend visually.

<figure>
  <img
    src={waveStructureTab}
    alt="WAVE Structure tab"
    style={waveToolbarGuideImageStyle}
  />
  <figcaption>
    WAVE Structure tab: left sidebar lists nested h1 and h2 headings and
    regions; the page shows h1 and h2 badges beside matching headings
  </figcaption>
</figure>

**Why this matters for structure:** Screen reader users often **navigate by heading** and **landmark**. A clear outline of regions and `h1`–`h6` headings lets users jump between parts of the page quickly, and the Structure view makes gaps, skipped levels, or missing regions easier to see than scanning the page by eye alone. W3C WAI rules, landmark and heading practice, WCAG success criteria, and **Spectrum Web Components** (`heading-level`, slots, and similar) live in the companion guide [Headings and landmarks](/docs/guides-accessibility-guides-headings-and-landmarks--readme).

### Order tab

The **Order** tab lists **navigable** elements in sequence with their **role** and **accessible name**, and can draw a **numbered path** on the page so you can compare **focus and reading order** to **visual placement**.

<figure>
  <img
    src={waveOrderTab}
    alt="WAVE Order tab"
    style={waveToolbarGuideImageStyle}
  />
  <figcaption>
    WAVE Order tab: styles off, numbered arrows show tab order across links and
    controls on a web page
  </figcaption>

Use it to check that users who move with **Tab** and users who follow the **announced order** in a screen reader get a sequence that matches your design. It does not replace keyboard testing, but it highlights mismatches between DOM order and layout.

### Contrast tab and color testing

The **Contrast** tab lets you set **foreground** and **background** colors (including with the color picker), read the **contrast ratio**, and see **WCAG AA/AAA** pass or fail for normal or large text. WAVE also notes that **gradients**, **filters**, and **transparency** may need **manual** checks with the eyedropper or other tools.

<figure>
  <img
    src={waveContrastTab}
    alt="WAVE Contrast tab"
    style={waveToolbarGuideImageStyle}
  />
  <figcaption>
    WAVE Contrast tab: foreground and background color inputs, contrast ratio
    8.59 to 1, WCAG AA and AAA pass, and Desaturate page link
  </figcaption>

**Desaturate page** renders the page in **grayscale** to reduce reliance on **hue** when checking hierarchy and meaning. It can surface cases where information or structure depends too much on color. It is **not** a full simulation of all types of color vision deficiency; dedicated **color blindness** browser extensions can simulate **specific** conditions (for example, protanopia, deuteranopia) in more detail.
