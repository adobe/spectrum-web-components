## Anatomy

A response status indicator consists of:

1. **Status row**: an indicator, status label, and optional disclosure control
2. **Step timeline**: optional execution steps that can be expanded for more detail

<Canvas of={ResponseStatusStories.Anatomy} />

## Options

### Statuses

The `status` attribute controls the lifecycle of the whole response:

- **`pending`**: the request is queued or preparing to start
- **`active`**: the response is actively working
- **`complete`**: the response finished successfully
- **`stopped`**: the user interrupted the response

Use `slot="label"` for the visible header text.

`pending` applies to the whole response before execution steps are available. Step rows begin at `active`, `complete`, or `stopped` once the timeline is shown.

<Canvas of={ResponseStatusStories.Statuses} />

### Steps

Add `swc-response-status-step` children to render an expandable timeline. Set `open` to show the timeline by default.

Each step uses:

- **`status`**: `active`, `complete`, or `stopped`
- **`slot="label"`**: step title
- **`slot="description"`**: detail text shown in the expanded timeline

All provided steps render in their authored order.

## Behaviors

### Handling events

`<swc-response-status>` emits **`swc-response-status-toggle`** when users open or close the step timeline. The event **bubbles** and is **composed**.

`event.detail.open` contains the next disclosure state. The component updates its own `open` state before emitting the event.

```js
status.addEventListener('swc-response-status-toggle', (event) => {
  const { open } = event.detail;
  console.log(open);
});
```

### Label update cadence

Header-label cadence is consumer-managed. The component rolls the header label whenever its text changes; your app decides when to update that text. If you stream rapid status updates, keep each visible label readable for about 1 second before replacing it.

The `LabelCadence` story demonstrates one consumer-side queue pattern.

## Accessibility

### Features

The `<swc-response-status>` element implements the following accessibility features:

- When no steps exist, non-final status labels are exposed with `role="status"` so updates are announced politely without moving focus
- When steps exist, the disclosure button uses `aria-expanded` and `aria-controls` to communicate panel state
- The step timeline uses `role="group"` with an accessible name from the `accessible-label` attribute (defaults to “Execution steps”)
- The ordered step list uses `role="list"` and each rendered step uses `role="listitem"`

### Best practices

- Keep step labels concise and action-oriented.
- Put required state in `slot="label"`; use `slot="description"` for extra context.
- Provide a meaningful `accessible-label` when multiple response-status components can appear on one screen.
